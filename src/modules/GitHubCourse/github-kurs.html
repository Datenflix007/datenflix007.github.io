Was ist Git?

Einführung · Versionskontrolle · Kernkonzepte

Git ist ein dezentrales Versionskontrollsystem (DVCS – Distributed Version Control System), das 2005 von Linus Torvalds entwickelt wurde – demselben Entwickler, der auch den Linux-Kernel geschrieben hat. Ziel war ein System, das schnell, sicher und vollständig verteilt arbeitet.

Was bedeutet Versionskontrolle?

Versionskontrolle bedeutet, dass jede Änderung an einer Datei protokolliert wird. Du kannst jederzeit zu einem früheren Zustand zurückkehren, verschiedene Entwicklungsstränge parallel führen und Änderungen mehrerer Personen zusammenführen – ohne dass sich Änderungen gegenseitig überschreiben.

Die drei Bereiche von Git

BereichBezeichnungBedeutung
Working DirectoryArbeitsbereichDeine Dateien auf der Festplatte – hier arbeitest du
Staging AreaIndexVorbereitungszone: du wählst, welche Änderungen in den nächsten Commit kommen
Repository.git-OrdnerDie Datenbank aller Commits und Branches – liegt lokal auf deiner Maschine

Dezentral vs. zentral

Bei zentralisierten Systemen wie SVN existiert die Historie nur auf einem Server. Bei Git dagegen hat jeder Entwickler eine vollständige Kopie des gesamten Repositories – inklusive aller Commits, Branches und der kompletten Historie. Das ermöglicht Offline-Arbeit und erhöht die Ausfallsicherheit erheblich.

Kernkonzepte

  • Commit – Ein Schnappschuss des Projekts zu einem bestimmten Zeitpunkt. Unveränderlich, durch einen SHA-1-Hash eindeutig identifizierbar.
  • Branch – Ein leichtgewichtiger Zeiger auf einen Commit. Branches ermöglichen parallele Entwicklungslinien (Features, Bugfixes, Experimente).
  • HEAD – Ein spezieller Zeiger, der anzeigt, welcher Commit oder Branch aktuell aktiv ist.
  • Merge – Das Zusammenführen zweier Branches. Git versucht Änderungen automatisch zu kombinieren.
  • Rebase – Alternative zu Merge: schreibt die Commit-Geschichte um, um eine lineare Historie zu erzeugen.
  • Tag – Ein fester Zeiger auf einen bestimmten Commit, typischerweise für Releases (z.B. v1.0.0).

Git ist das mit Abstand meistgenutzte Versionskontrollsystem der Welt. Laut Stack Overflow Developer Survey 2024 verwenden über 93 % aller Entwicklerinnen und Entwickler Git in ihrem Arbeitsalltag.

Git Datenmodell

# Git speichert Objekte in einer einfachen Schlüssel-Wert-Datenbank. # Es gibt vier Objekttypen: # 1. blob – der Inhalt einer einzelnen Datei # 2. tree – ein Verzeichnis (enthält blobs und weitere trees) # 3. commit – ein Schnappschuss + Metadaten (Autor, Zeit, Eltern-Commit) # 4. tag – annotierter Zeiger auf einen Commit # Jeder Commit zeigt auf genau einen tree (das Root-Verzeichnis) # und einen oder mehrere Eltern-Commits. # (commit A) ──► (commit B) ──► (commit C) ◄── HEAD, main # └──► (commit D) ◄── feature/login # SHA-1 Hash eines beliebigen Commits anzeigen: git log --oneline # Internen Typ eines Objekts inspizieren: git cat-file -t # Inhalt eines Objekts anzeigen: git cat-file -p

Was ist GitHub?

Einführung · Plattform · Features

GitHub ist eine webbasierte Hosting-Plattform für Git-Repositories. Gegründet 2008, seit 2018 Teil von Microsoft, ist GitHub heute die weltweit größte Entwicklungsplattform mit über 100 Millionen Entwicklerinnen und Entwicklern und mehr als 420 Millionen öffentlichen Repositories.

GitHub ≠ Git

Ein häufiges Missverständnis: Git und GitHub sind nicht dasselbe. Git ist das Werkzeug (lokal, Open Source, kostenlos). GitHub ist ein Dienst, der Git-Repositories in der Cloud hostet und mit zusätzlichen Kollaborationsfunktionen anreichert. Du könntest Git auch vollständig ohne GitHub nutzen – und umgekehrt gibt es andere Hosting-Angebote für Git-Repositories (GitLab, Bitbucket, Gitea, etc.).

Der Pull Request Workflow

Das zentrale Kollaborationsmodell auf GitHub basiert auf Pull Requests (PRs). Statt direkt auf den Hauptbranch zu pushen, entwickelst du auf einem Feature-Branch, öffnest dann einen PR und bittest Teammitglieder um ein Code-Review. Erst nach Freigabe wird der Branch gemergt. Dieser Workflow ist heute Industriestandard.

1
Fork oder Branch erstellen

Bei eigenen Repos: neuen Branch. Bei fremden Repos: Fork des Repositories erstellen.

2
Änderungen committen und pushen

Entwickle dein Feature lokal, committe und pushe den Branch zu GitHub.

3
Pull Request öffnen

Auf GitHub einen PR erstellen: Beschreibung hinzufügen, Reviewer zuweisen.

4
Code Review

Teammitglieder kommentieren, fordern Änderungen an oder genehmigen.

5
Merge

Nach Genehmigung wird der Branch in main gemergt. Branch kann danach gelöscht werden.

GitHub-Konten sind für Open-Source-Projekte und persönliche Repositories kostenlos. Für Teams ab einer bestimmten Größe und mit erweiterten CI/CD-Minuten gibt es kostenpflichtige Pläne (Team, Enterprise).

Erstes Repository auf GitHub

# Szenario: Du hast ein lokales Projekt und willst es auf GitHub hochladen. # 1. Lokales Repo initialisieren (falls noch nicht geschehen) git init git add . git commit -m "Initial commit" # 2. Auf GitHub: Neues Repository anlegen (ohne README, ohne .gitignore) # Danach erhältst du eine URL wie: https://github.com/deinname/mein-projekt.git # 3. Remote hinzufügen und pushen git remote add origin https://github.com/deinname/mein-projekt.git git branch -M main git push -u origin main # -u setzt origin/main als Upstream-Tracking-Branch. # Danach reicht einfach: git push und git pull

GitLab im Vergleich

Einführung · GitHub vs. GitLab vs. Bitbucket

GitLab ist eine vollständige DevOps-Plattform, die neben Git-Hosting auch CI/CD, Container Registry, Security-Scanning und vieles mehr in einem einzigen Produkt vereint. Im Gegensatz zu GitHub kann GitLab auch selbst gehostet (on-premise) betrieben werden – ein entscheidender Vorteil in datenschutzsensiblen Umgebungen wie Forschung oder Behörden.

Wann welche Plattform?

  • GitHub – Standardwahl für Open-Source-Projekte, persönliche Portfolios, Community-Beiträge. Größtes Ecosystem, beste Sichtbarkeit.
  • GitLab – Empfehlenswert wenn du alles in einer Plattform haben möchtest, Self-Hosting benötigst (z.B. Universität, Behörde) oder tiefe CI/CD-Integration brauchst.
  • Bitbucket – Sinnvoll wenn du bereits im Atlassian-Ökosystem arbeitest (Jira, Confluence) und enge Integration brauchst.

Terminologische Unterschiede

Das Grundprinzip ist auf allen Plattformen identisch – Git bleibt Git. Die Terminologie unterscheidet sich jedoch geringfügig: was GitHub einen Pull Request nennt, heißt bei GitLab Merge Request. Was GitHub GitHub Actions nennt, heißt bei GitLab GitLab CI/CD Pipelines. Die Konzepte dahinter sind dieselben.

Tipp für Einsteiger: Starte mit GitHub. Es hat die größte Community, die meisten Tutorials und die beste Integration mit Tools wie VS Code. Du kannst dasselbe lokale Git-Repository jederzeit auf eine andere Plattform spiegeln – die Umstiegskosten sind niedrig.

Repository zu beiden Plattformen spiegeln

# Ein Repository gleichzeitig auf GitHub und GitLab pushen # Schritt 1: Zwei Remotes konfigurieren git remote add github https://github.com/deinname/mein-projekt.git git remote add gitlab https://gitlab.com/deinname/mein-projekt.git # Schritt 2: Zu beiden gleichzeitig pushen (via push-all Alias) git remote add all https://github.com/deinname/mein-projekt.git git remote set-url --add --push all https://github.com/deinname/mein-projekt.git git remote set-url --add --push all https://gitlab.com/deinname/mein-projekt.git git push all main # Pusht zu beiden gleichzeitig # Alle konfigurierten Remotes anzeigen: git remote -v

Installation & Konfiguration

Terminal · Erste Schritte · SSH-Keys

Bevor du Git nutzen kannst, muss es auf deinem System installiert und mit deiner Identität konfiguriert werden. Diese Einstellungen werden in jeder Commit-Nachricht eingebettet und sind für die Zusammenarbeit auf GitHub wichtig.

Installation

  • macOS: brew install git (Homebrew) oder Xcode Command Line Tools
  • Ubuntu/Debian: sudo apt install git
  • Fedora/RHEL: sudo dnf install git
  • Windows: git-scm.com/download/win – Git Bash inklusive

Pflichtschritte nach der Installation

Diese zwei Befehle müssen einmalig nach der Installation ausgeführt werden. Git speichert sie global in ~/.gitconfig.

SSH-Keys einrichten

Statt bei jedem Push dein Passwort einzugeben, empfiehlt sich SSH-Authentifizierung. Du erzeugst ein Schlüsselpaar: der private Key bleibt lokal, der öffentliche Key wird auf GitHub hinterlegt. Ab dann authentifiziert sich Git automatisch.

Sicherheitshinweis: Teile deinen privaten SSH-Key (id_ed25519) niemals mit jemandem und committe ihn unter keinen Umständen in ein Repository. Füge *.pem, id_rsa* und id_ed25519 zur .gitignore hinzu.

Die .gitignore Datei

Eine .gitignore-Datei im Wurzelverzeichnis deines Projekts legt fest, welche Dateien und Ordner Git ignorieren soll. Typische Kandidaten: Build-Artefakte, Abhängigkeitsordner (node_modules/), Umgebungsvariablen (.env), IDE-Konfigurationen (.idea/, .vscode/) und Betriebssystem-Dateien (.DS_Store, Thumbs.db).

Installation & Konfiguration

Gute Commit-Nachrichten

Eine gute Commit-Nachricht beschreibt warum eine Änderung vorgenommen wurde, nicht nur was. Weit verbreitet ist der Conventional Commits-Standard: feat: Anmeldeformular hinzugefügt, fix: Null-Pointer in UserService behoben, docs: README aktualisiert, refactor: Datenbankabfrage optimiert.

Branching & Merging

Terminal · Parallele Entwicklung · Merge-Strategien

Branches sind das mächtigste Konzept von Git. Ein Branch ist technisch gesehen nur ein 41-Byte großer Zeiger auf einen Commit – das Erstellen und Wechseln von Branches kostet daher nahezu keine Zeit. Branches ermöglichen es, Features, Bugfixes und Experimente vollständig voneinander isoliert zu entwickeln.

Branch-Befehle

BefehlBedeutung

git branch <name>Neuen Branch erstellen (aber nicht wechseln)
git checkout <name>Zu einem Branch wechseln
git checkout -b <name>Branch erstellen und sofort wechseln
git switch <name>Modernere Alternative zu checkout (Git 2.23+)
git switch -c <name>Branch erstellen und wechseln (Switch-Variante)
git branch -d <name>Branch löschen (nur wenn gemergt)
git branch -D <name>Branch löschen erzwingen
git push origin --delete <name>Remote-Branch löschen

Merge-Strategien

StrategieBefehlWann verwenden
Fast-Forwardgit merge <branch>Kein Divergenz – Zeiger wird einfach vorgerückt. Sauberste Option.
Merge Commitgit merge --no-ff <branch>Erzwingt einen Merge-Commit auch bei Fast-Forward. Bewahrt Branch-Struktur.

Branch-Namenskonventionen

  • main oder master – Hauptbranch (stabiler, produktiver Code)
  • develop – Integrationsbranch für Features (bei Git-Flow)
  • feature/login-seite – Neue Funktionalität
  • bugfix/null-pointer-userservice – Fehlerbehebung
  • hotfix/kritischer-sicherheitsfehler – Dringender Fix auf Produktion
  • release/v1.2.0 – Release-Vorbereitung

Rebase vs. Merge: Rebase schreibt Commits-Hashes um und ist daher nur für lokale, noch nicht geteilte Branches geeignet. Regel: Niemals einen Branch rebasen, der bereits auf einem Remote liegt und von anderen genutzt wird.

Feature-Branch Workflow

Nach dem Merge auf GitHub: Remote-Branch aufräumen git checkout main git pull origin main # Enthält jetzt den gemergten Feature-Branch git branch -d feature/benutzer-profil # Lokalen Branch löschen git remote prune origin # Gelöschte Remote-Branches aus lokalem Cache entfernen # ─── REBASE: Lineale Historie ─────────────────────────────────────── # Szenario: main hat neue Commits, dein Feature-Branch hinkt hinterher git checkout feature/benutzer-profil git rebase main # Git hebt deine Commits auf, aktualisiert die Basis auf main, # und wendet deine Commits danach erneut an. # Falls Konflikte auftreten: # 1. Konflikte manuell lösen git add git rebase --continue # Rebase abbrechen (zum ursprünglichen Zustand zurück) git rebase --abort

GitHub CLI (gh)

Die GitHub CLI (gh) erlaubt es, GitHub-Funktionen direkt vom Terminal aus zu nutzen: Pull Requests erstellen, Issues anlegen, Repositories forken – ohne den Browser öffnen zu müssen. Installation via brew install gh (macOS), sudo apt install gh (Linux) oder dem Windows-Installer.

Fork-Workflow

Ein Fork ist eine persönliche Kopie eines fremden Repositories auf deinem GitHub-Account. Du forkt, um zu einem Projekt beizutragen, zu dem du keine direkten Push-Rechte hast. Änderungen werden via Pull Request an das Original-Repository zurückgeschickt.

Konflikte lösen

Terminal · Merge-Konflikte · Strategien · Tools

Ein Merge-Konflikt entsteht, wenn zwei Branches dieselbe Stelle in derselben Datei unterschiedlich geändert haben und Git nicht automatisch entscheiden kann, welche Version die richtige ist. Konflikte sind normal – kein Grund zur Panik. Git markiert die betroffenen Stellen, du löst sie manuell, und der Merge wird abgeschlossen.

Wie sieht ein Konflikt aus?

Git fügt Konfliktmarker direkt in die Datei ein. Der Abschnitt zwischen <<<<<<< HEAD und ======= zeigt die Version des aktuellen Branches. Der Abschnitt zwischen ======= und >>>>>>> zeigt die Version des eingehenden Branches.

Konflikte Schritt für Schritt lösen

1
Konflikte identifizieren

git status zeigt alle Dateien mit dem Status "both modified"

2
Datei öffnen und Marker verstehen

Suche nach <<<<<<<. Lies beide Versionen.

3
Entscheidung treffen

Behalte eine Version, die andere, oder kombiniere beide. Entferne alle Konfliktmarker.

4
Datei als gelöst markieren

git add <datei> – teilt Git mit, dass der Konflikt gelöst ist

5
Merge abschließen

git commit – Git erstellt den Merge-Commit automatisch

Merge-Tools

  • VS Code – Beste Wahl: zeigt Konflikte grafisch mit "Accept Current/Incoming/Both" Buttons
  • git mergetool – Öffnet konfiguriertes Diff-Tool (vimdiff, meld, kdiff3)
  • GitHub Web-Editor – Für einfache Konflikte direkt im Browser lösbar

Konflikte vermeiden: Regelmäßig pullen (git pull --rebase), kurze Feature-Branches, klare Aufgabenteilung im Team. Wenige große Commits führen zu mehr Konflikten als viele kleine, atomare Commits.

Fortgeschrittene Befehle

Terminal · Reset · Revert · Cherry-pick · Bisect · Worktree

Mit zunehmender Erfahrung werden Situationen auftreten, die über den Standardworkflow hinausgehen: ein falscher Commit wurde gemacht, ein Fehler soll aus der History entfernt werden, oder nur ein einzelner Commit aus einem anderen Branch wird benötigt. Die folgenden Befehle geben dir die volle Kontrolle.

Änderungen rückgängig machen

BefehlWas passiertHistorie
git revert <hash>Erstellt einen neuen Commit, der den alten rückgängig machtBleibt erhalten ✓
git reset --soft HEAD~1Letzten Commit rückgängig, Änderungen bleiben gestagtWird geändert
git reset --mixed HEAD~1Letzten Commit rückgängig, Änderungen bleiben unstagedWird geändert
git reset --hard HEAD~1Letzten Commit + alle Änderungen unwiderruflich löschenWird gelöscht ⚠

Nützliche Befehle

BefehlBedeutung
git cherry-pick <hash>Einen einzelnen Commit aus einem anderen Branch übernehmen
git bisect startBinäre Suche nach dem Commit, der einen Bug eingeführt hat
git reflogAlle Bewegungen von HEAD anzeigen – Rettungsanker wenn Commits "verloren" scheinen
git tag v1.0.0Lightweight Tag für aktuellen Commit
git tag -a v1.0.0 -m "Release"Annotierter Tag mit Nachricht
git push --tagsAlle Tags zum Remote pushen
git worktree add ../hotfix hotfix/xyzZweites Working Directory für anderen Branch – kein stash nötig
git shortlog -snCommits pro Autor (für Beitragsstatistiken)

Vorsicht mit git reset --hard und git push --force: Beide Befehle können Daten dauerhaft verlieren lassen. git reset --hard niemals auf Commits anwenden, die bereits gepusht wurden. Stattdessen git revert verwenden – das ist die sichere, kollaborationsfreundliche Alternative.

GitHub Desktop – Einführung

GitHub Desktop · Installation · Oberfläche · Für wen?

GitHub Desktop ist eine kostenlose, von GitHub entwickelte grafische Oberfläche (GUI) für Git. Sie richtet sich an Entwicklerinnen und Entwickler, die eine visuelle Alternative zum Terminal bevorzugen – ohne dabei auf professionellen Workflow verzichten zu müssen. GitHub Desktop ist auf macOS und Windows verfügbar.

Für wen ist GitHub Desktop geeignet?

  • Einsteiger, die Git-Konzepte visuell verstehen wollen
  • Entwickler, die häufig zwischen vielen Branches wechseln
  • Teams, die Pull Requests schnell erstellen und reviewen
  • Alle, die Diffs und den Staging-Prozess visuell bevorzugen

Installation

Download unter desktop.github.com. Nach der Installation meldet man sich mit dem GitHub-Konto an – SSH-Keys und Konfiguration werden automatisch eingerichtet. Kein manuelles git config nötig.

Wichtige Bereiche der Oberfläche

BereichFunktion

Current BranchAktueller Branch, Branch wechseln oder neu erstellen
Fetch / Pull / PushSynchronisation mit GitHub – zeigt Anzahl ausstehender Commits
Changes TabAlle geänderten Dateien – Checkboxen zum Stagen, Diff-Ansicht rechts
History TabVollständige Commit-Liste mit Diff für jeden Commit
Commit-BoxSummary (Pflicht) und Description (optional) eingeben, dann Commit

Terminal oder Desktop?

GitHub Desktop und das Terminal schließen sich nicht aus – viele Entwickler nutzen beide parallel. GitHub Desktop ist schnell für den Standardworkflow (add, commit, push, PR erstellen). Das Terminal ist unverzichtbar für fortgeschrittene Operationen (rebase, bisect, reflog, cherry-pick). Es empfiehlt sich, beide zu kennen.

GitHub Desktop speichert alle Repositories in einer Liste und ermöglicht es, per Tastenkürzel Strg/Cmd+K schnell zwischen ihnen zu wechseln. Über Repository → Open in Terminal wechselst du jederzeit ins Terminal mit bereits gesetztem Repository-Pfad.

Workflow mit GitHub Desktop

GitHub Desktop · Branch · Commit · Pull Request · Konflikte

GitHub Desktop deckt den kompletten täglichen Workflow visuell ab. Folgend ist der empfohlene Ablauf für ein Feature von Anfang bis zum gemergten Pull Request – ohne eine einzige Zeile Terminal.

Feature-Workflow komplett in GitHub Desktop

1
Repository aktualisieren

Oben rechts auf Fetch origin klicken. Falls neue Commits vorliegen: Pull origin.

2
Neuen Branch erstellen

Current Branch → New Branch. Name nach Konvention: feature/mein-feature.

3
Änderungen vornehmen

In VS Code oder einem anderen Editor arbeiten. Geänderte Dateien erscheinen automatisch im Changes-Tab.

4
Dateien selektiv stagen

Checkboxen neben den Dateien: aktiviert = wird gestagt. Per Rechtsklick auf einzelne Zeilen im Diff können sogar einzelne Zeilen gestagt werden.

5
Commit erstellen

Summary (Pflicht, max. 72 Zeichen) und optionale Description eingeben. Commit to feature/... klicken.

6
Branch pushen & Pull Request erstellen

Publish branch (erster Push) → danach erscheint Create Pull Request, das direkt zu GitHub führt.

7
Konflikte in Desktop lösen

Bei Konflikten zeigt Desktop welche Dateien betroffen sind und bietet an, sie in VS Code oder dem integrierten Editor zu öffnen.

Diff-Ansicht verstehen

Im Changes-Tab zeigt GitHub Desktop den Diff jeder geänderten Datei. Rote Zeilen wurden entfernt, grüne Zeilen wurden hinzugefügt. Mit der Split-Ansicht (Schalter rechts oben im Diff) können alte und neue Version nebeneinander verglichen werden.

Undone: Mit Cmd/Ctrl+Z kann der letzte Commit direkt rückgängig gemacht werden – solange er noch nicht gepusht wurde. GitHub Desktop hält alle letzten Commits als undoable bereit.

Typischer Desktop-Workflow

# GitHub Desktop: Workflow-Äquivalente als Terminal-Befehle # Desktop: "Fetch origin" / "Pull origin" git fetch origin git pull origin main # Desktop: "New Branch" → feature/mein-feature git switch -c feature/mein-feature # Desktop: Checkboxen in Changes Tab = git add git add src/index.html git add src/style.css # (oder git add -p für zeilenweises Stagen) # Desktop: Summary + "Commit to branch" git commit -m "feat: Startseite gestaltet" # Desktop: "Publish Branch" git push -u origin feature/mein-feature # Desktop: "Create Pull Request" → öffnet github.com/repo/compare/feature/... gh pr create --title "feat: Startseite" --body "Erster Entwurf der Startseite" # Desktop: Branch nach Merge aufräumen git checkout main git pull origin main git branch -d feature/mein-feature

Deployment – Schritt für Schritt

GitHub Pages · Manuell aktivieren · Diese Seite deployen

Das Deployment auf GitHub Pages ist in wenigen Schritten abgeschlossen. Sobald Pages aktiviert ist, veröffentlicht GitHub jede Änderung am konfigurierten Branch automatisch – ohne weiteres Zutun.

Schritt-für-Schritt: Diese Seite deployen

Auf github.com: New Repository. Name wählen (z.B. github-kurs). Öffentlich setzen.

2
HTML-Datei hinzufügen

Die index.html dieser Kurseite ins Wurzelverzeichnis des Repositories legen und pushen.

3
GitHub Pages aktivieren

Repository → Settings → Pages → Source: Deploy from a branch → Branch: main, Ordner: / (root) → Save.

4
Warten und URL aufrufen

GitHub deployt innerhalb von ca. 30–60 Sekunden. Die URL erscheint unter Settings → Pages: https://nutzername.github.io/github-kurs

5
Zukünftige Updates

Jeder Push zu main löst automatisch ein Re-Deployment aus. Keine weitere Konfiguration nötig.

Deployment-Status prüfen

Wichtig: GitHub Pages ist für öffentlich zugängliche, statische Inhalte gedacht. Keine Passwörter, API-Keys, private Daten oder serverseitige Logik. Sensible Daten gehören in Umgebungsvariablen auf einem echten Server, nicht in ein öffentliches Repository.

Automatisierung mit GitHub Actions

GitHub Actions · CI/CD · YAML Workflows · Pages automatisch bauen

GitHub Actions ist Githubs integriertes CI/CD-System. Workflows werden als YAML-Dateien im Verzeichnis .github/workflows/ definiert und durch Events ausgelöst: ein Push, ein Pull Request, ein Schedule (Cronjob) oder manuell. Actions können Tests ausführen, Builds erstellen, Code linten und Websites auf GitHub Pages deployen.

Nützliche Actions aus dem Marketplace

  • actions/checkout@v4 – Repository auschecken (fast immer erster Step)
  • actions/setup-node@v4 – Node.js-Version installieren
  • actions/setup-python@v5 – Python-Version installieren
  • actions/upload-pages-artifact@v3 – Build-Ergebnis für Pages vorbereiten
  • actions/deploy-pages@v4 – Auf GitHub Pages deployen
  • actions/cache@v4 – Dependencies cachen (schnellere Builds)

Workflows für diese Kursseite

Da diese Kursseite eine einzelne, eigenständige HTML-Datei ist, braucht sie keinen Build-Schritt. Der einfachste Ansatz ist das direkte Deployment aus dem main-Branch. Für komplexere Projekte (z.B. mit einem Build-Tool wie Vite oder einem Static Site Generator) wird stattdessen ein zweistufiger Workflow genutzt: erst bauen, dann deployen.

GitHub stellt pro Monat 2000 kostenlose Actions-Minuten für private Repositories zur Verfügung (Stand 2025). Für öffentliche Repositories sind Actions vollständig kostenlos und unbegrenzt.
