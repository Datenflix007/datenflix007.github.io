Du bist Codex und arbeitest in einem bestehenden statischen Website-Projekt namens „AlltagsLabor“.

Ziel:
Setze das Projekt so um, dass die bestehende GitHub Page Experimente weiterhin aus JSON-Dateien lädt, aber künftig aus einem GitHub-Datenrepo statt aus GitLab. Ergänze außerdem eine sichere Einpflegemaske, über die ausgewählte Benutzer mit unpersönlichen Zugangsdaten neue Experimente einreichen können. Die Einreichung darf nicht direkt aus dem Browser in das Repo schreiben. Stattdessen soll sie über eine Serverless-Function laufen, die einen GitHub Actions Workflow triggert. Der Workflow validiert den Eintrag und erstellt einen Pull Request, der die passende JSON-Datei ergänzt.

Ausgangslage:

* Es gibt eine bestehende `index.html`.
* Die Seite lädt aktuell Experimente aus einem GitLab-Datenrepo.
* In der Datei existieren aktuell Konstanten ähnlich:
  `DATA_REPO_API = "https://gitlab.com/api/v4/projects/Datenflix007%2Falltagslabordata/repository/files/"`
  und Sprachdateien:
  `_experiments_de.json`, `_experiments_eng.json`, `_experiments_fr.json`, `_experiments_ru.json`, `_experiments_uk.json`.
* Die bestehende Anzeige, Suche, Filterung, Detailansicht, Druckfunktion und PDF-Funktion sollen erhalten bleiben.
* Die Struktur der Experimentobjekte soll kompatibel bleiben:

  * `title`
  * `shortDescription`
  * `subject`
  * `gradeLevel`
  * `schoolType`
  * `steps`
  * Schritte können z. B. `type: "text"`, `type: "aufgabe"`, `type: "merksatz"`, `type: "image"` oder `type: "audio"` haben.

Aufgaben:

1. GitHub-Datenquelle für die bestehende Website

* Ersetze die GitLab-Ladelogik in `index.html` durch GitHub-Raw-Dateien.
* Nutze als konfigurierbare Basis:
  `https://raw.githubusercontent.com/Datenflix007/alltagslabordata/main/`
* Die Sprachdateien sollen weiterhin heißen:

  * `_experiments_de.json`
  * `_experiments_eng.json`
  * `_experiments_fr.json`
  * `_experiments_ru.json`
  * `_experiments_uk.json`
* Passe Funktionen wie `gitlabFileUrl`, `dataUrlForLanguage`, `resolveAsset` und `rewriteMediaPaths` entsprechend an.
* Benenne Funktionen neutral um, z. B. `repoFileUrl` statt `gitlabFileUrl`.
* Die Besucheransicht soll funktional gleich bleiben.

2. Neue Einpflegemaske
   Erstelle eine neue Datei:
   `einpflegen.html`

Die Seite soll optisch zur bestehenden `index.html` passen:

* gleiche Grundfarben
* ähnliche Topbar
* gleicher Stil für Buttons, Felder und Panels
* responsives Layout

Die Maske soll folgende Felder enthalten:

* Zugangscode oder Passwort
* Sprache: `de`, `eng`, `fr`, `ru`, `uk`
* Titel
* Kurzbeschreibung
* Fach
* Klassenstufe
* Schulform
* Mehrere Schritte, dynamisch hinzufügbar

  * Schritt-Typ: `text`, `aufgabe`, `merksatz`, `image`, `audio`
  * Inhalt
  * optionale Beschreibung bei Medien
* Vorschau des erzeugten JSON-Objekts
* Button „Experiment einreichen“

Beim Absenden soll aus den Feldern ein JSON-Objekt im bestehenden Format erzeugt werden.

3. Clientseitige Validierung
   Baue in `einpflegen.html` sinnvolle Validierung ein:

* Zugangscode darf nicht leer sein.
* Sprache muss eine der erlaubten Sprachkennungen sein.
* Titel, Kurzbeschreibung, Fach, Klassenstufe und Schulform dürfen nicht leer sein.
* Es muss mindestens ein Schritt vorhanden sein.
* Jeder Schritt braucht Typ und Inhalt.
* Bei `image` und `audio` soll der Inhalt als Pfad oder URL verstanden werden.
* Zeige verständliche Fehlermeldungen direkt auf der Seite.
* Nach erfolgreichem Absenden soll eine klare Erfolgsmeldung angezeigt werden.

4. Serverless-Function-Vorlage
   Lege einen Ordner an:
   `serverless/`

Erstelle darin eine Beispielimplementierung für Cloudflare Workers:
`serverless/cloudflare-worker.js`

Diese Function soll:

* nur POST-Anfragen erlauben
* JSON entgegennehmen
* den Zugangscode gegen ein Secret prüfen, z. B. `SUBMISSION_PASSWORD`
* das Experiment grob validieren
* einen GitHub Actions Workflow über die GitHub API starten
* dafür `workflow_dispatch` verwenden
* die Nutzdaten an den Workflow übergeben:

  * Sprache
  * Experimentobjekt
  * optional Einreicherkennung, wenn vorhanden
* das GitHub-Token aus einem Secret lesen, z. B. `GITHUB_TOKEN`
* Repo-Owner und Repo-Name über Variablen konfigurierbar machen:

  * `GITHUB_OWNER`
  * `GITHUB_REPO`
  * `GITHUB_WORKFLOW_FILE`, z. B. `submit-experiment.yml`
* keine Secrets im Frontend verwenden
* CORS sauber setzen, sodass die GitHub Page die Function aufrufen kann
* verständliche JSON-Antworten zurückgeben

Wichtig:
Das Frontend darf niemals ein GitHub-Token enthalten.

5. Frontend-Anbindung
   In `einpflegen.html` soll der Endpoint konfigurierbar sein:
   `const SUBMISSION_ENDPOINT = "https://example-worker.example.workers.dev";`

Der Code soll die Formulardaten per `fetch` als POST an diesen Endpoint senden.

6. GitHub Actions Workflow
   Erstelle:
   `.github/workflows/submit-experiment.yml`

Der Workflow soll per `workflow_dispatch` gestartet werden können und Inputs erhalten:

* `language`
* `experiment_json`
* `submitter`

Der Workflow soll:

* das passende Sprachfile bestimmen:

  * `de` → `_experiments_de.json`
  * `eng` → `_experiments_eng.json`
  * `fr` → `_experiments_fr.json`
  * `ru` → `_experiments_ru.json`
  * `uk` → `_experiments_uk.json`
* abbrechen, wenn eine nicht erlaubte Sprache übergeben wird
* prüfen, ob die Datei existiert
* JSON parsen und validieren
* das neue Experiment an das bestehende Array anhängen
* die JSON-Datei formatiert speichern
* einen neuen Branch erstellen, z. B.:
  `submission/<datum>-<slug>`
* einen Pull Request gegen `main` erstellen
* PR-Titel:
  `Neues Experiment: <Titel>`
* PR-Beschreibung mit:

  * Sprache
  * Fach
  * Klassenstufe
  * Schulform
  * Einreicherkennung
  * Hinweis, dass der Eintrag automatisch aus der Einpflegemaske erzeugt wurde

Nutze nach Möglichkeit:

* `actions/checkout`
* Node.js für Validierung und Dateiänderung
* `peter-evans/create-pull-request` oder eine vergleichbare etablierte Action

7. Validierungsskript
   Erstelle ein Node-Skript:
   `scripts/append-experiment.mjs`

Das Skript soll:

* Sprache und Experiment-JSON aus Umgebungsvariablen lesen
* die erlaubte Sprachdatei bestimmen
* das Experiment validieren
* bestehende Datei laden
* prüfen, ob sie ein Array enthält
* neues Experiment anhängen
* Datei mit 2 Leerzeichen Einrückung zurückschreiben
* bei Fehlern mit verständlicher Fehlermeldung abbrechen

Validierungsregeln:

* `title`, `shortDescription`, `subject`, `gradeLevel`, `schoolType` müssen Strings und nicht leer sein
* `steps` muss ein Array mit mindestens einem Eintrag sein
* jeder Schritt muss `type` und `content` besitzen
* erlaubte Schritt-Typen:

  * `text`
  * `aufgabe`
  * `merksatz`
  * `image`
  * `audio`
* unbekannte Zusatzfelder sollen nicht automatisch gelöscht werden

8. Dokumentation
   Erstelle oder ergänze eine Datei:
   `README.md`

Dokumentiere knapp:

* wie die Besucheransicht funktioniert
* wie die GitHub-Datenquelle aufgebaut ist
* wie neue Experimente über `einpflegen.html` eingereicht werden
* warum ein Serverless-Backend nötig ist
* welche Secrets gesetzt werden müssen:

  * `SUBMISSION_PASSWORD`
  * `GITHUB_TOKEN`
  * `GITHUB_OWNER`
  * `GITHUB_REPO`
  * `GITHUB_WORKFLOW_FILE`
* wie der Cloudflare Worker veröffentlicht wird
* wie der GitHub Workflow arbeitet
* dass neue Einträge als Pull Request landen und erst nach Review produktiv werden

9. Sicherheitsanforderungen

* Kein GitHub-Token im Frontend.
* Keine direkten Commits in `main` durch externe Nutzer.
* Neue Experimente sollen als Pull Request landen.
* Serverseitige Validierung ist Pflicht.
* Clientseitige Validierung ist nur Komfort, nicht Sicherheit.
* Der gemeinsame Zugangscode ist nur eine einfache Zugangshürde, keine echte Benutzerverwaltung.
* HTML-Inhalte aus Formularfeldern nicht blind ausführen. Mindestens dokumentieren, dass bei öffentlicher Nutzung zusätzlich Sanitizing nötig ist. Wenn möglich, einfache Sanitizing-Funktion einbauen oder gefährliche Tags wie `<script>`, `<iframe>`, `<object>`, Eventhandler-Attribute wie `onclick`, und `javascript:`-URLs entfernen.

10. Kompatibilität

* Bestehende Funktionen der `index.html` dürfen nicht kaputtgehen.
* Das bestehende Experimentformat muss weiter funktionieren.
* Die Seite soll lokal auch ohne Buildsystem laufen.
* Verwende plain HTML, CSS und JavaScript, sofern möglich.
* Für das Node-Skript im Workflow darf Node.js verwendet werden.

11. Ergebnis
    Bitte ändere oder erstelle die nötigen Dateien:

* `index.html`
* `einpflegen.html`
* `serverless/cloudflare-worker.js`
* `.github/workflows/submit-experiment.yml`
* `scripts/append-experiment.mjs`
* `README.md`

Gib am Ende eine kurze Zusammenfassung:

* welche Dateien geändert wurden
* welche Secrets noch gesetzt werden müssen
* welche URL in `SUBMISSION_ENDPOINT` eingetragen werden muss
* wie man den ersten Testeintrag einreicht
