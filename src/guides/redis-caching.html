Redis Caching Guide

Warum Cache-Speicher Anwendungen schneller macht, wann er Probleme verursacht und wie du Redis mit praktischen Beispielen lokal ausprobierst.

Redis ist ein sehr schneller In-Memory-Datenspeicher. Er wird häufig als Cache eingesetzt: Daten, die sonst immer wieder aus Datenbanken, Dateisystemen, APIs oder aufwendigen Berechnungen entstehen würden, werden kurzzeitig im Arbeitsspeicher abgelegt und bei der nächsten Anfrage direkt von dort gelesen.

Ein Cache ersetzt keine saubere Datenhaltung. Er ist eine zusätzliche Schicht zwischen Anwendung und eigentlicher Datenquelle. Richtig eingesetzt reduziert er Antwortzeiten, Datenbanklast und Kosten. Falsch eingesetzt liefert er veraltete Daten, verbraucht unnötig Speicher und macht Fehler schwerer nachvollziehbar.

Schneller lesen

Häufig abgefragte Ergebnisse kommen aus dem RAM statt aus Datenbank, Datei oder externer API.

Last reduzieren

Wiederholte Queries, teure Berechnungen und API-Aufrufe werden nicht unnötig mehrfach ausgeführt.

Bewusst begrenzen

TTL, Invalidierung und Speichergrenzen entscheiden, ob ein Cache hilfreich oder gefährlich wird.

Warum nicht immer direkt aus dem Massenspeicher lesen?

Massenspeicher wie SSDs, Festplatten, Datenbanken oder entfernte APIs sind deutlich langsamer als Arbeitsspeicher. Der Unterschied liegt nicht nur in der reinen Lesezeit, sondern auch in allem, was rundherum passiert: Netzwerkverkehr, SQL-Parsing, Indexsuche, Dateisystemzugriffe, Authentifizierung, Serialisierung und Sperren.

QuelleTypische EigenschaftProblem bei häufiger Nutzung
DatenbankDauerhafte, konsistente DatenViele identische Queries belasten CPU, Indizes und Verbindungen.
DateisystemPersistente DateienViele kleine Lesezugriffe erzeugen Overhead und blockieren I/O.
Externe APIDaten von FremdsystemenLatenz, Rate Limits, Ausfälle und Kosten pro Anfrage.
BerechnungAktuell erzeugtes ErgebnisCPU-Zeit wird für gleiche Ergebnisse mehrfach verschwendet.
Redis CacheSchneller Zugriff im RAMDaten können veralten und Speicher ist begrenzt.

Wenn eine Produktseite, ein Dashboard oder eine Rangliste tausendmal pro Minute abgerufen wird, ist es oft unnötig, jedes Mal dieselben Tabellen zu joinen oder dieselbe JSON-Antwort aus einer API zu holen. Der Cache speichert das Ergebnis für eine kurze Zeit, zum Beispiel 30 Sekunden, 5 Minuten oder eine Stunde.

Was Redis besonders gut kann

  • Sehr schnelle Lese- und Schreibzugriffe: Redis hält Daten primär im Arbeitsspeicher.
  • TTL pro Schlüssel: Einträge können automatisch nach einer festgelegten Zeit ablaufen.
  • Einfache Datenstrukturen: Strings, Hashes, Listen, Sets, Sorted Sets und Streams.
  • Atomare Operationen: Viele Befehle sind einzeln konsistent, zum Beispiel INCR für Zähler.
  • Sprachunterstützung: Es gibt Clients für Node.js, Python, Java, C#, PHP, Go und viele weitere Sprachen.
  • Praktische Zusatzrollen: Rate Limiting, Session Store, Warteschlangen, Locks und Pub/Sub.

Für Caching ist besonders wichtig: Redis kann einem Schlüssel direkt eine Ablaufzeit geben. Dadurch verschwinden Cache-Einträge automatisch, ohne dass die Anwendung später mühsam alte Einträge suchen und löschen muss.

Wann Caching sinnvoll ist

Teure Anfrage

Die Datenquelle ist langsam, teuer oder CPU-intensiv, aber das Ergebnis ändert sich nicht ständig.

Viele gleiche Zugriffe

Viele Benutzer fragen dieselben Produktdaten, Dashboards, Feeds oder Statistiken ab.

Externe API

Die Antwort hat spürbare Latenz, Rate Limits, Ausfallrisiko oder Kosten pro Anfrage.

Aktualität hat Spielraum

Eine Verzögerung von Sekunden oder Minuten ist fachlich akzeptabel.

Typische Beispiele

Use CaseCache-DauerWarum?
Startseiten-Feed30 bis 120 SekundenViele Leser, kleine Verzögerung akzeptabel.
Produktdetails5 bis 30 MinutenProduktdaten ändern sich seltener als sie gelesen werden.
API-Antwort einer Wetter-API10 bis 30 MinutenExterne Anfrage kostet Zeit und ist nicht sekündlich nötig.
Benutzerrechte1 bis 5 MinutenSchnellere Checks, aber Änderungen sollen bald greifen.
Rate Limit CounterFenstergröße, z. B. 60 SekundenRedis kann Zähler sehr effizient erhöhen und verfallen lassen.

Wann du keinen Cache verwenden solltest

Caching ist nicht automatisch besser. Es lohnt sich vor allem dann, wenn wiederholte Zugriffe teuer sind und leichte Verzögerungen bei der Aktualität akzeptiert werden können.

Transaktionen

Kontostände, Zahlungen oder Lagerbestände dürfen nicht versehentlich veraltet verarbeitet werden.

Selten gelesen

Wenn ein Wert fast nie erneut abgefragt wird, verschwendet ein Cache nur Speicher.

Zu große Objekte

Riesige JSON-Dokumente können Redis-Speicher schnell füllen und Netzwerkverkehr erhöhen.

Unklare Invalidierung

Wenn niemand weiß, wann ein Eintrag veraltet ist, wird der Cache zur Fehlerquelle.

Faustregel: Erst messen, dann cachen. Eine seltene 3-ms-Query braucht meist keinen Redis-Cache. Eine 800-ms-Anfrage, die tausendfach wiederholt wird, ist ein guter Kandidat.

Zentrale Begriffe

BegriffBedeutung
Cache HitDer gesuchte Wert liegt im Cache und kann direkt zurückgegeben werden.
Cache MissDer Wert fehlt im Cache; die Anwendung muss ihn aus der Originalquelle holen.
TTLTime to Live: Zeit, nach der ein Cache-Eintrag automatisch gelöscht wird.
InvalidierungGezieltes Löschen oder Aktualisieren veralteter Cache-Einträge.
EvictionRedis entfernt Einträge, weil der konfigurierte Speicher knapp wird.
Cache StampedeViele Prozesse merken gleichzeitig einen Cache Miss und berechnen denselben Wert parallel neu.

Erste Redis-Befehle

docker exec -it redis-cache-demo redis-cli

TTL page:home
DEL page:home

EXPIRE views:article:42 3600

EX 60 bedeutet: Dieser Schlüssel läuft nach 60 Sekunden ab. TTL zeigt die verbleibende Lebenszeit in Sekunden an.

Cache-Aside Pattern

Das häufigste Muster heißt Cache Aside. Die Anwendung fragt zuerst Redis. Wenn der Wert fehlt, liest sie aus der Datenbank oder API, speichert das Ergebnis in Redis und liefert es zurück.

1Anfrage kommt rein
2Redis nach cache:key fragen
3Bei Hit direkt antworten
4Bei Miss Originalquelle laden
5Mit TTL speichern und antworten

Dieses Pattern ist einfach, robust und passt gut zu HTTP-APIs, Produktseiten, Profilen, Statistiken und berechneten Antworten.

Node.js Beispiel zum Ausprobieren

Dieses Beispiel simuliert eine langsame Datenquelle. Beim ersten Aufruf wartet die API künstlich 1,2 Sekunden. Danach kommt die Antwort aus Redis.

server.js

import express from "express";
import { createClient } from "redis";

const app = express();
const redis = createClient({ url: "redis://localhost:6379" });

redis.on("error", (error) => {
  console.error("Redis error:", error);
});

await redis.connect();

function wait(ms) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

async function loadProductFromDatabase(id) {
  await wait(1200);
  return {
    id,
    name: `Produkt ${id}`,
    price: 19.99,
    loadedAt: new Date().toISOString()
  };
}

app.get("/products/:id", async (req, res) => {
  const id = req.params.id;
  const cacheKey = `product:${id}`;

  const cached = await redis.get(cacheKey);
  if (cached) {
    return res.json({
      source: "redis-cache",
      data: JSON.parse(cached)
    });
  }

  const product = await loadProductFromDatabase(id);
  await redis.set(cacheKey, JSON.stringify(product), {
    EX: 30
  });

  res.json({
    source: "slow-database",
    data: product
  });
});

app.delete("/products/:id/cache", async (req, res) => {
  const deleted = await redis.del(`product:${req.params.id}`);
  res.json({ deleted });
});

app.listen(3000, () => {
  console.log("Server läuft auf http://localhost:3000");
});

Cache-Keys sinnvoll benennen

Redis ist ein Key-Value-Store. Gute Schlüsselnamen sind entscheidend, damit Cache-Einträge nachvollziehbar bleiben.

SchlüsselBedeutung
product:42Produkt mit ID 42
user:7:permissionsBerechtigungen von Benutzer 7
dashboard:teacher:2026-07Monatsdashboard für Lehrkräfte
api:weather:jenaGecachte Antwort einer Wetter-API für Jena
ratelimit:login:192.0.2.10Login-Versuche einer IP-Adresse

Vermeide unklare Schlüssel wie data1, tmp oder cache. Ein Schlüssel sollte erkennen lassen, welche Art von Daten er enthält und wodurch er eindeutig wird.

Invalidierung: Das schwierige Thema

Der schwerste Teil beim Caching ist nicht das Speichern, sondern das richtige Entfernen veralteter Daten. Ein Cache ist nur hilfreich, solange die Anwendung weiß, wann ein Wert neu geladen werden muss.

Strategie 1: TTL verwenden

await redis.set("product:42", JSON.stringify(product), { EX: 300 });

Nach 5 Minuten läuft der Eintrag aus. Das ist einfach und gut für Daten, bei denen leichte Verzögerungen akzeptabel sind.

Strategie 2: Nach Schreibvorgängen löschen

async function updateProduct(id, changes) {
  await database.products.update(id, changes);
  await redis.del(`product:${id}`);
}

Wenn ein Produkt geändert wird, löscht die Anwendung den passenden Cache-Eintrag. Die nächste Leseanfrage lädt den aktuellen Wert neu.

Strategie 3: Versionierte Keys

const version = await redis.get("products:version");
const cacheKey = `products:v${version}:list:page:${page}`;

Bei großen Listen kann es einfacher sein, eine Version hochzuzählen, statt viele einzelne Keys zu löschen. Alte Keys laufen später durch TTL aus.

Cache Stampede verhindern

Ein Cache Stampede entsteht, wenn ein beliebter Cache-Eintrag abläuft und sehr viele Anfragen gleichzeitig denselben teuren Wert neu berechnen. Die Datenbank wird dann trotz Cache plötzlich stark belastet.

Einfache Gegenmaßnahmen

  • TTL leicht zufällig machen, damit nicht alle Keys gleichzeitig ablaufen.
  • Teure Werte bereits im Hintergrund erneuern, bevor sie ablaufen.
  • Für sehr teure Berechnungen einen kurzen Redis-Lock verwenden.

TTL mit Zufallsanteil

const ttl = 300 + Math.floor(Math.random() * 60);
await redis.set(cacheKey, JSON.stringify(value), { EX: ttl });

Einfacher Lock mit Redis

const lockKey = `lock:${cacheKey}`;
const lockAcquired = await redis.set(lockKey, "1", {
  NX: true,
  EX: 10
});

if (lockAcquired) {
  try {
    const fresh = await loadExpensiveValue();
    await redis.set(cacheKey, JSON.stringify(fresh), { EX: 300 });
    return fresh;
  } finally {
    await redis.del(lockKey);
  }
}

await new Promise((resolve) => setTimeout(resolve, 150));
const cachedAfterWait = await redis.get(cacheKey);

NX bedeutet: Setze den Lock nur, wenn der Schlüssel noch nicht existiert. Die kurze TTL sorgt dafür, dass ein Lock auch bei einem Absturz nicht dauerhaft hängen bleibt.

Rate Limiting mit Redis

Redis eignet sich sehr gut für Zähler, weil INCR atomar ist. Das folgende Beispiel begrenzt eine IP-Adresse auf 10 Anfragen pro Minute.

async function checkRateLimit(ip) {
  const key = `ratelimit:${ip}`;
  const count = await redis.incr(key);

  if (count === 1) {
    await redis.expire(key, 60);
  }

  return {
    allowed: count <= 10,
    remaining: Math.max(0, 10 - count)
  };
}

Das ist kein vollständiges Security-System, aber ein gutes Grundmuster für Login-Versuche, Kontaktformulare oder öffentliche API-Endpunkte.

Sessions in Redis

Bei Webanwendungen mit mehreren Serverinstanzen sollten Sessions nicht nur im lokalen Arbeitsspeicher eines einzelnen Servers liegen. Sonst verliert ein Benutzer seine Session, wenn die nächste Anfrage bei einer anderen Instanz landet.

session:user:7 -> {
  "userId": 7,
  "csrfToken": "...",
  "createdAt": "2026-07-04T12:00:00Z"
}

Redis kann hier als zentraler Session Store dienen. Wichtig sind kurze, klare Ablaufzeiten und ein bewusster Umgang mit sensiblen Daten. Sessions sollten nicht unbegrenzt leben.

Speichergrenzen und Eviction

Redis läuft im Arbeitsspeicher. Deshalb muss ein Produktivsystem eine Speichergrenze und eine passende Räumstrategie haben. Ohne Grenze kann Redis im schlimmsten Fall zu viel RAM verbrauchen.

# Beispiel in redis.conf
maxmemory 512mb
maxmemory-policy allkeys-lru
PolicyVerhalten
noevictionRedis lehnt neue Schreibvorgänge ab, wenn der Speicher voll ist.
allkeys-lruRedis entfernt ungefähr die am längsten nicht genutzten Keys.
volatile-lruRedis entfernt nur Keys mit TTL nach LRU-Logik.
allkeys-randomRedis entfernt zufällige Keys.
volatile-ttlRedis entfernt bevorzugt Keys, die bald ablaufen.

Für reine Caches ist allkeys-lru häufig sinnvoll. Für Systeme, in denen Redis auch wichtige Nicht-Cache-Daten enthält, muss man deutlich vorsichtiger planen.

Use Case: API-Antwort cachen mit Node.js

Geeignet für fremde APIs, die langsam sind, Rate Limits haben oder Kosten pro Anfrage verursachen.

import { createClient } from "redis";

const redis = createClient({ url: "redis://localhost:6379" });
await redis.connect();

async function getWeather(city) {
  const key = `api:weather:${city.toLowerCase()}`;
  const cached = await redis.get(key);

  if (cached) {
    return JSON.parse(cached);
  }

  const response = await fetch(`https://api.example.com/weather?city=${city}`);
  const weather = await response.json();

  await redis.set(key, JSON.stringify(weather), { EX: 600 });
  return weather;
}

Use Case: Datenbank-Query cachen mit Python

Geeignet für Profilseiten, Produktdetails, Dashboards oder alle Daten, die häufiger gelesen als geschrieben werden.

import json
import redis

cache = redis.Redis(host="localhost", port=6379, decode_responses=True)


def get_product(product_id, db):
    key = f"product:{product_id}"
    cached = cache.get(key)

    if cached:
        return json.loads(cached)

    product = db.fetch_one(
        "SELECT id, name, price FROM products WHERE id = ?",
        [product_id]
    )

    cache.setex(key, 300, json.dumps(product))
    return product


def update_product(product_id, changes, db):
    db.update("products", product_id, changes)
    cache.delete(f"product:{product_id}")

Use Case: Spring Boot Service mit Redis Cache

In Java-Projekten wird Redis häufig über Spring Cache eingebunden. Der Service-Code bleibt dadurch sehr kurz.

// build.gradle
dependencies {
    implementation "org.springframework.boot:spring-boot-starter-cache"
    implementation "org.springframework.boot:spring-boot-starter-data-redis"
}
// application.properties
spring.data.redis.host=localhost
spring.data.redis.port=6379
spring.cache.type=redis
spring.cache.redis.time-to-live=300000
import org.springframework.cache.annotation.CacheEvict;
import org.springframework.cache.annotation.Cacheable;
import org.springframework.stereotype.Service;

@Service
public class ProductService {
    private final ProductRepository repository;

    public ProductService(ProductRepository repository) {
        this.repository = repository;
    }

    @Cacheable(value = "products", key = "#id")
    public Product findById(long id) {
        return repository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("Product not found"));
    }

    @CacheEvict(value = "products", key = "#id")
    public Product update(long id, ProductUpdate update) {
        Product product = repository.findById(id).orElseThrow();
        product.setName(update.name());
        product.setPrice(update.price());
        return repository.save(product);
    }
}

Use Case: PHP Cache-Aside mit Predis

Geeignet für klassische PHP-Anwendungen, CMS-Erweiterungen oder kleine API-Backends.

composer require predis/predis
<?php

require __DIR__ . "/vendor/autoload.php";

$redis = new Predis\Client("tcp://127.0.0.1:6379");

function getArticle(int $id, PDO $pdo, Predis\Client $redis): array
{
    $key = "article:" . $id;
    $cached = $redis->get($key);

    if ($cached !== null) {
        return json_decode($cached, true);
    }

    $stmt = $pdo->prepare("SELECT id, title, body FROM articles WHERE id = ?");
    $stmt->execute([$id]);
    $article = $stmt->fetch(PDO::FETCH_ASSOC);

    $redis->setex($key, 300, json_encode($article));
    return $article;
}

Use Case: Go HTTP Handler mit Redis

Dieses Beispiel verwendet go-redis und cached eine JSON-Antwort.

go mod init redis-go-demo
go get github.com/redis/go-redis/v9
package main

import (
    "context"
    "encoding/json"
    "net/http"
    "time"

    "github.com/redis/go-redis/v9"
)

var ctx = context.Background()

var rdb = redis.NewClient(&redis.Options{
    Addr: "localhost:6379",
})

type Product struct {
    ID       string    `json:"id"`
    Name     string    `json:"name"`
    LoadedAt time.Time `json:"loadedAt"`
}

func productHandler(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    key := "product:" + id

    cached, err := rdb.Get(ctx, key).Result()
    if err == nil {
        w.Header().Set("Content-Type", "application/json")
        w.Write([]byte(`{"source":"redis-cache","data":` + cached + `}`))
        return
    }

    time.Sleep(1200 * time.Millisecond)
    product := Product{ID: id, Name: "Produkt " + id, LoadedAt: time.Now().UTC()}
    bytes, _ := json.Marshal(product)

    rdb.Set(ctx, key, bytes, 30*time.Second)

    w.Header().Set("Content-Type", "application/json")
    w.Write([]byte(`{"source":"slow-database","data":` + string(bytes) + `}`))
}

func main() {
    http.HandleFunc("GET /products/{id}", productHandler)
    http.ListenAndServe(":8080", nil)
}

Use Case: Login-Versuche begrenzen in Python

Ein einfaches Rate-Limit schützt Login-Formulare vor zu vielen Versuchen in kurzer Zeit.

def allow_login_attempt(ip: str) -> bool:
    key = f"login-attempts:{ip}"
    attempts = cache.incr(key)

    if attempts == 1:
        cache.expire(key, 300)

    return attempts <= 5

Use Case: Session speichern in Node.js

Sessions sollten immer eine Ablaufzeit haben. Dadurch verschwinden alte Sitzungen automatisch.

import crypto from "node:crypto";

async function createSession(userId) {
  const sessionId = crypto.randomUUID();
  const key = `session:${sessionId}`;

  await redis.set(key, JSON.stringify({
    userId,
    createdAt: new Date().toISOString()
  }), {
    EX: 60 * 60 * 8
  });

  return sessionId;
}

async function getSession(sessionId) {
  const session = await redis.get(`session:${sessionId}`);
  return session ? JSON.parse(session) : null;
}

async function deleteSession(sessionId) {
  await redis.del(`session:${sessionId}`);
}

Use Case: Pub/Sub für Live-Updates

Redis kann Nachrichten an mehrere Prozesse verteilen. Das ist nützlich für einfache Live-Updates, Chat-Prototypen oder interne Events.

// publisher.js
import { createClient } from "redis";

const publisher = createClient();
await publisher.connect();

await publisher.publish("notifications", JSON.stringify({
  type: "USER_CREATED",
  userId: 42
}));

await publisher.quit();
// subscriber.js
import { createClient } from "redis";

const subscriber = createClient();
await subscriber.connect();

await subscriber.subscribe("notifications", (message) => {
  const event = JSON.parse(message);
  console.log("Event erhalten:", event);
});

Use Case: Teure Berechnung cachen

Auch ohne Datenbank lohnt sich ein Cache, wenn eine Berechnung langsam ist und mit gleichen Parametern mehrfach gebraucht wird.

async function cachedComputation(input) {
  const key = `compute:sha256:${input}`;
  const cached = await redis.get(key);

  if (cached) {
    return JSON.parse(cached);
  }

  const result = await veryExpensiveComputation(input);
  await redis.set(key, JSON.stringify(result), { EX: 3600 });
  return result;
}

Produktionshinweise

Absichern

Redis nicht ungeschützt ins öffentliche Internet stellen. Netzwerkregeln, Passwort und bei Bedarf TLS verwenden.

Keys ordnen

Klare Prefixes nutzen, zum Beispiel cache:, session: und ratelimit:.

TTL setzen

Nicht jeden Cache-Eintrag unbegrenzt speichern. Ablaufzeiten gehören zum Cache-Design.

Überwachen

Hit Rate, Speicherverbrauch, Anzahl der Keys, Evictions und Latenz beobachten.

Ausfall einplanen

Die Anwendung sollte bei Redis-Ausfall langsamer, aber möglichst weiterhin korrekt arbeiten.

Sensible Daten prüfen

Personenbezogene Daten, Tokens und Sessions brauchen kurze TTLs und ein klares Schutzkonzept.

Mini-Experiment: Cache-Wirkung messen

Mit PowerShell kannst du grob sehen, ob der zweite Aufruf schneller wird. Starte vorher das Node-Beispiel.

Measure-Command { Invoke-RestMethod http://localhost:3000/products/42 }
Measure-Command { Invoke-RestMethod http://localhost:3000/products/42 }

Unter macOS oder Linux funktioniert ein einfacher Vergleich mit time:

time curl -s http://localhost:3000/products/42
time curl -s http://localhost:3000/products/42

Der erste Aufruf sollte wegen der simulierten Datenbank langsamer sein. Der zweite Aufruf kommt aus Redis und sollte deutlich schneller antworten.

Checkliste für eigene Projekte

  • Welche Anfrage ist wirklich langsam oder teuer?
  • Wie oft wird dieselbe Antwort erneut benötigt?
  • Wie alt darf die Antwort maximal sein?
  • Welcher Cache-Key beschreibt die Daten eindeutig?
  • Wann wird der Key gelöscht oder erneuert?
  • Was passiert, wenn Redis nicht erreichbar ist?
  • Wie viel RAM darf der Cache maximal nutzen?
  • Welche Daten dürfen aus Datenschutzgründen nicht oder nur sehr kurz gecacht werden?

Kurzfazit

Redis-Caching ist besonders stark, wenn viele gleiche oder ähnliche Anfragen auf teure Datenquellen treffen. Es verhindert unnötige Datenbank- und API-Zugriffe, macht Anwendungen reaktionsschneller und hilft bei Lastspitzen. Gleichzeitig ist ein Cache immer eine zusätzliche Fehlerquelle: Er muss ablaufen, gelöscht, überwacht und fachlich verstanden werden.

Die beste Praxis ist deshalb nicht „alles cachen“, sondern gezielt cachen: Messbare Engpässe identifizieren, sinnvolle TTLs wählen, klare Keys verwenden und bei Änderungen gezielt invalidieren.
