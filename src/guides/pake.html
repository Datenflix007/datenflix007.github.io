Desktop Apps Rust & Tauri

Pake Guide

Mit Pake fast jede Webseite in eine schlanke Desktop-Anwendung für Windows, macOS oder Linux verpacken.

Pake ist ein Open-Source-Werkzeug, das aus einer URL oder einer lokalen HTML-Datei eine Desktop-Anwendung erzeugt. Statt eine komplette Browser-Laufzeit wie bei vielen Electron-Apps mitzuliefern, nutzt Pake Tauri, Rust und die WebView-Komponenten des jeweiligen Betriebssystems.

Das Ergebnis ist keine komplett neu programmierte native App, sondern ein nativer Container um eine Weboberfläche. Genau das ist für viele Webdienste praktisch: Chat-Tools, Dashboards, Lernplattformen, interne Admin-Oberflächen oder persönliche Web-Apps bekommen ein eigenes Fenster, ein eigenes Icon und können wie normale Programme gestartet werden.

Was Pake macht

  • Eine Webseite wird in ein Desktop-Fenster gepackt.
  • Die App erhält einen Namen, ein Icon und eine Start-URL.
  • Fenstergröße, Zoom, Dark Mode, Tray-Verhalten und Navigation können angepasst werden.
  • Pake erzeugt installierbare Pakete für Windows, macOS und Linux.
  • Die App bleibt technisch eine WebView-App: Sie rendert HTML, CSS und JavaScript wie ein Browserfenster.

Ein typischer Aufruf sieht so aus:

pake https://github.com --name GitHub

Danach liegt die erzeugte Desktop-App im aktuellen Arbeitsordner. Beim ersten Build dauert es länger, weil die Rust-/Tauri-Werkzeuge eingerichtet und kompiliert werden. Spätere Builds sind meist deutlich schneller.

Theoretischer Hintergrund

Web-App, Desktop-App und Wrapper

Eine normale Webseite läuft im Browser. Eine klassische Desktop-App wird für ein Betriebssystem gebaut und direkt installiert. Pake liegt dazwischen: Es nimmt eine Web-App und verpackt sie in eine Desktop-Hülle. Diese Hülle startet ein Fenster, lädt die Zielseite und stellt Betriebssystem-Funktionen wie Icon, Menü, Fensterzustand oder Installer bereit.

Das ist besonders nützlich, wenn eine Anwendung ohnehin als Web-App existiert. Statt die komplette Oberfläche in C#, Swift, Kotlin oder C++ neu zu schreiben, wird die bestehende Weboberfläche weiterverwendet.

Warum Tauri statt Electron?

Electron-Apps bringen in der Regel eine eigene Chromium- und Node.js-Laufzeit mit. Das ist bequem, aber schwer: Jede App enthält viel Infrastruktur, die auf dem System oft ohnehin schon vorhanden ist. Tauri geht anders vor. Es nutzt die WebView des Betriebssystems und koppelt sie mit einem kleinen nativen Rust-Kern.

AnsatzIdeeFolge
Browser-TabWebseite bleibt im normalen BrowserEinfach, aber nicht app-artig
ElectronApp bringt Chromium selbst mitSehr kompatibel, aber oft groß
Tauri/PakeApp nutzt System-WebView und RustKompakter und ressourcenschonender

Welche WebView wird verwendet?

  • Windows: Microsoft Edge WebView2.
  • macOS: WebKit, also die Safari-nahe System-WebView.
  • Linux: WebKitGTK.

Das erklärt auch, warum eine Pake-App auf verschiedenen Betriebssystemen minimal unterschiedlich wirken kann. Sie nutzt nicht überall exakt denselben Browserkern, sondern den jeweils vorhandenen System-Unterbau.

Was Rust dabei macht

Rust ist die Sprache, in der Tauri seine native Anwendungsschicht baut. Diese Schicht verwaltet Fenster, Menüs, Build-Ziele, Dateisystemzugriffe, Icons, Installer und optionale Systemfunktionen. Für die eigentliche Webseite bleibt weiterhin HTML, CSS und JavaScript zuständig.

Als Nutzer von Pake musst du normalerweise kein Rust programmieren. Rust wird gebraucht, damit der native App-Container kompiliert werden kann.

Wann Pake sinnvoll ist

  • Eine Webseite soll ein eigenes App-Fenster bekommen.
  • Ein internes Dashboard soll ohne Browser-Chaos geöffnet werden.
  • Eine Lernplattform, ein Wiki oder ein Tool soll wie eine Desktop-App wirken.
  • Du willst eine kleine Alternative zu schweren Electron-Wrappern bauen.
  • Du brauchst eine schnelle Verpackung, aber keine tiefen nativen Betriebssystem-Integrationen.

Wann Pake nicht ideal ist

  • Die Webseite funktioniert nur in einem ganz bestimmten Browser.
  • Sehr viele native APIs werden gebraucht, etwa komplexe Dateisystem-, Bluetooth- oder Hardwarezugriffe.
  • Die Zielseite blockiert WebViews, fremde User-Agents oder eingebettete Login-Flows.
  • Du willst eine offline-first App, obwohl die Webseite zwingend einen Server braucht.
  • Du musst App-Store-Verteilung, Signierung und Auto-Updates professionell betreiben. Das geht, ist aber mehr als ein Einzeiler.

Installation der Dependencies

Für die normale Pake-CLI brauchst du vor allem Node.js, pnpm oder npm, Rust und die systemnahen Tauri-Abhängigkeiten. Die offiziellen Pake-Hinweise empfehlen Node.js 22 oder neuer; Node 18+ kann funktionieren, aber für neue Setups ist Node 22 die sauberere Wahl.

1. Node.js installieren

Windows mit WinGet

winget install --id OpenJS.NodeJS.LTS --exact
node -v
npm -v

macOS mit Homebrew

brew install node
node -v
npm -v

Linux mit Paketmanager

# Debian/Ubuntu, falls die Distribution eine passende Node-Version anbietet
sudo apt update
sudo apt install nodejs npm

# Version prüfen
node -v
npm -v

Wenn die Distribution nur eine alte Node-Version liefert, ist ein Versionsmanager wie fnm, nvm oder ein offizielles NodeSource-Paket sinnvoll.

2. pnpm aktivieren

Pake empfiehlt die Installation über pnpm. Moderne Node-Versionen bringen corepack mit:

corepack enable
pnpm -v

Falls das nicht funktioniert, kann pnpm auch über npm installiert werden:

npm install -g pnpm
pnpm -v

3. Rust installieren

Windows

winget install --id Rustlang.Rustup --exact
rustc --version
cargo --version

Unter Windows sollte die MSVC-Toolchain verwendet werden. Falls Rust bereits installiert war, kann sie so gesetzt werden:

rustup default stable-msvc

macOS und Linux

curl --proto '=https' --tlsv1.2 https://sh.rustup.rs -sSf | sh
rustc --version
cargo --version

Nach der Installation muss das Terminal oft neu gestartet werden, damit cargo im Suchpfad verfügbar ist.

4. Systemabhängigkeiten für Tauri

Windows

Für Windows werden die Microsoft C++ Build Tools und Microsoft Edge WebView2 benötigt. Auf Windows 10 und Windows 11 ist WebView2 häufig bereits vorhanden. Die Build Tools installierst du über den Visual-Studio-Installer mit der Auswahl Desktop development with C++.

# Prüfen, ob Node, Rust und Cargo erreichbar sind
node -v
npm -v
rustc --version
cargo --version

macOS

Für reine Desktop-Builds genügen normalerweise die Xcode Command Line Tools:

xcode-select --install
rustc --version
node -v

Debian/Ubuntu

sudo apt update
sudo apt install libwebkit2gtk-4.1-dev \
  build-essential \
  curl \
  wget \
  file \
  libxdo-dev \
  libssl-dev \
  libayatana-appindicator3-dev \
  librsvg2-dev

Fedora

sudo dnf install webkit2gtk4.1-devel \
  openssl-devel \
  curl \
  wget \
  file \
  libappindicator-gtk3-devel \
  librsvg2-devel \
  libxdo-devel

sudo dnf group install "c-development"

Arch Linux

sudo pacman -Syu
sudo pacman -S --needed \
  webkit2gtk-4.1 \
  base-devel \
  curl \
  wget \
  file \
  openssl \

5. Pake CLI installieren

pnpm install -g pake-cli
pake --version
pake --help

Alternative mit npm:

npm install -g pake-cli
pake --version

Ohne globale Installation kann Pake auch mit npx gestartet werden:

npx pake-cli https://github.com --name GitHub

Erste App bauen

Wechsle in einen Ordner, in dem die fertige App landen soll, und starte:

mkdir pake-apps
cd pake-apps
pake https://github.com --name GitHub

Pake versucht automatisch, ein passendes Icon von der Webseite zu holen. Die erzeugte Datei ist je nach Betriebssystem ein anderes Paketformat, zum Beispiel .exe, .msi, .dmg, .app, .deb, .rpm oder .AppImage.

Fenstergröße festlegen

pake https://calendar.google.com --name Kalender --width 1280 --height 900

Eigenes Icon verwenden

pake https://example.com --name BeispielApp --icon ./icon.png

Pake kann lokale und entfernte Icons verwenden. Plattformabhängig werden passende Formate erzeugt oder erwartet, etwa .ico für Windows, .icns für macOS und .png für Linux.

Debug-Modus einschalten

pake https://example.com --name BeispielDebug --debug

Der Debug-Modus ist hilfreich, wenn du die Entwicklerwerkzeuge brauchst, um CSS, JavaScript, Login-Probleme oder WebView-Unterschiede zu untersuchen.

Die vollständige Liste hängt von der installierten Pake-Version ab. Deshalb lohnt sich immer:

pake --help

Beispiel 1: Eigene Webseite als Desktop-App

Angenommen, du willst deine eigene Webseite als Desktop-App starten:

pake https://datenflix007.github.io --name Datenflix \
  --width 1200 \
  --height 800 \
  --min-width 900 \
  --min-height 600

Das ist sinnvoll, wenn du eine Webseite häufig als eigenes Arbeitsfenster offen haben willst.

Beispiel 2: Lokales HTML-Projekt verpacken

Pake kann auch lokale HTML-Dateien verpacken. Das ist praktisch für kleine Tools, die komplett aus HTML, CSS und JavaScript bestehen:

pake ./index.html --name MeinLokalesTool --width 1000 --height 720

Wenn dein lokales Projekt mehrere Dateien verwendet, müssen relative Pfade zu CSS, JavaScript und Bildern weiterhin stimmen. Am besten testest du die Seite zuerst direkt im Browser.

Beispiel 3: Dashboard mit interner Navigation

Viele Dienste leiten beim Login auf weitere Subdomains oder SSO-Anbieter weiter. Mit --safe-domain kann man solche Domains in der App halten:

pake https://dashboard.example.com --name FirmenDashboard \
  --safe-domain example.com,login.example.com \
  --width 1440 \
  --height 950

Ohne solche Regeln kann es passieren, dass ein Link oder Login-Redirect im normalen Browser statt in der Pake-App geöffnet wird.

Beispiel 4: Kleine Always-on-top App

Für Timer, Statusseiten oder Notizzettel kann ein kleines Fenster praktisch sein:

pake https://timer.example.com --name Timer \
  --width 420 \
  --height 620 \
  --always-on-top

Beispiel 5: AppImage oder Debian-Paket bauen

Unter Linux können Ausgabeziele explizit gesetzt werden:

pake https://github.com --name GitHub --targets appimage
pake https://github.com --name GitHub --targets deb

Welche Targets sinnvoll sind, hängt von Distribution, Architektur und installierten Build-Werkzeugen ab.

Typischer Workflow

  1. Webseite im normalen Browser testen.
  2. Prüfen, ob Login, Cookies, Downloads und Navigation funktionieren.
  3. Ersten Pake-Build mit --debug erzeugen.
  4. Fenstergröße, Zoom und Icon anpassen.
  5. Bei Login- oder Redirect-Problemen --safe-domain oder --internal-url-regex setzen.
  6. Fertige App ohne Debug-Modus neu bauen.
  7. Auf dem Zielbetriebssystem testen.
# Entwicklungsbuild
pake https://app.example.com --name BeispielApp --debug --width 1280 --height 860

# Finaler Build
pake https://app.example.com --name BeispielApp --width 1280 --height 860

Häufige Probleme

pake wird nicht gefunden

Der globale npm-/pnpm-Binärordner ist nicht im PATH oder das Terminal wurde nach der Installation nicht neu gestartet.

pnpm env
npm config get prefix
pake --version

Rust oder Cargo wird nicht gefunden

Rust wurde installiert, aber das Terminal kennt den neuen Suchpfad noch nicht. Terminal neu starten und dann prüfen:

rustc --version
cargo --version

Windows-Build scheitert bei C++

Installiere die Microsoft C++ Build Tools und wähle im Installer Desktop development with C++. Danach Terminal neu starten.

Linux-Build scheitert bei WebKitGTK

Die Tauri-Systempakete fehlen oder passen nicht zur Distribution. Installiere die WebKitGTK-, OpenSSL-, AppIndicator-, Build- und SVG-Pakete für deine Distribution.

Login öffnet sich im Browser

Setze zusätzliche interne Domains:

pake https://app.example.com --name BeispielApp --safe-domain example.com,auth.example.com

Die Webseite sieht anders aus als im Browser

Das kann an der WebView liegen. Windows, macOS und Linux nutzen unterschiedliche Engines. Prüfe mit --debug, ob CSS, JavaScript, User-Agent oder blockierte APIs die Ursache sind.

Die erzeugte App ist nicht signiert

Pake kann schnell lokale Apps erzeugen. Professionelle Verteilung ist ein eigener Schritt: Unter macOS sind Code Signing und Notarization relevant, unter Windows Zertifikate und SmartScreen-Reputation, unter Linux Paketformat und Repository-Verteilung.

Sicherheit und Datenschutz

  • Verpacke nur Webseiten, denen du vertraust.
  • Eine Pake-App ist nicht automatisch sicherer als die Webseite selbst.
  • Login-Daten, Cookies und Sessions liegen im Kontext der App-WebView.
  • Bei fremden Webdiensten gelten weiterhin deren Datenschutz- und Nutzungsbedingungen.
  • Für Schul-, Uni- oder Unternehmenskontexte sollte geklärt werden, ob eine inoffizielle Desktop-Verpackung erlaubt ist.

Lizenz kurz erklärt

Pake selbst steht unter GPL-3.0. Das Projekt beschreibt zusätzlich eine Output Exception: Apps, die du mit Pake erzeugst, gehören dir und können unabhängig verteilt werden. Wenn du Pake selbst forkst und als eigenes Produkt weiterentwickelst, greifen dagegen die Lizenzpflichten für diesen Fork.
