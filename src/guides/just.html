Command Runner Automatisierung

just Guide

Projektbefehle in einem justfile sammeln, dokumentieren und unter Windows, Linux und macOS zuverlässig ausführen.

just ist ein plattformübergreifender Command Runner. Wiederkehrende Befehle werden als sogenannte Recipes in einer Datei namens justfile abgelegt und anschließend über kurze, einheitliche Kommandos gestartet. So wird aus docker compose -f compose.dev.yml up --build beispielsweise einfach just dev.

Die Syntax ist von make inspiriert, aber just ist bewusst kein Build-System. Recipes werden immer als auszuführende Aufgaben behandelt. Dateizeitstempel, implizite Build-Regeln und .PHONY-Deklarationen spielen keine Rolle.

Warum just verwenden?

  • Wichtige Projektbefehle liegen versioniert direkt im Repository.
  • Alle Teammitglieder verwenden dieselben Befehle und Optionen.
  • Recipes können Parameter, Abhängigkeiten und Umgebungsvariablen besitzen.
  • .env-Dateien lassen sich automatisch laden.
  • Recipes können Shell, PowerShell, Python, Node.js oder andere Interpreter verwenden.
  • just findet das justfile auch beim Aufruf aus einem Unterordner.
  • Die Aufgabenliste dient gleichzeitig als ausführbare Projektdokumentation.

Installation

Windows mit WinGet

winget install --id Casey.Just --exact
just --version

Alternativ stehen unter Windows unter anderem Scoop und Chocolatey zur Verfügung:

scoop install just
choco install just

macOS

brew install just

Linux

# Debian 13 / Ubuntu 24.04 und neuere passende Distributionen
sudo apt install just

# Fedora
sudo dnf install just

# Arch Linux
sudo pacman -S just

# Alpine Linux
sudo apk add just

Plattformübergreifend mit Rust

cargo install just

Danach Installation und verfügbaren Suchpfad prüfen:

just --version
just --help

Das erste justfile

Lege im Wurzelverzeichnis des Projekts eine Datei namens justfile ohne Dateiendung an:

# Verfügbare Aufgaben anzeigen
default:
    @just --list

# Entwicklungsserver starten
dev:
    npm run dev

# Tests ausführen
test:
    npm test

# Produktions-Build erzeugen
build:
    npm run build

Recipes werden mit ihrem Namen aufgerufen:

just          # führt die erste Recipe aus
just dev
just test
just build

Einrückung ist Teil der Syntax. Befehlszeilen müssen eingerückt sein; vier Leerzeichen sind eine gut lesbare Konvention. Mit just --fmt kann das justfile automatisch formatiert werden.

Recipes auflisten und beschreiben

Kommentare direkt vor einer Recipe werden mit just --list als Beschreibung angezeigt, wenn sie mit # beginnen:

# Entwicklungsumgebung starten
dev:
    npm run dev

# Alle automatisierten Prüfungen ausführen
check:
    npm run lint
    npm test
    npm run build
just --list
just --summary
BefehlBedeutung
just --listRecipes mit Parametern und Beschreibungen anzeigen
just --summaryNur die Namen aller Recipes ausgeben
just --show devDefinition einer einzelnen Recipe anzeigen
just --dumpDas ausgewertete justfile ausgeben

Befehlsausgabe steuern

Normalerweise zeigt just jede Befehlszeile vor ihrer Ausführung an. Ein vorangestelltes @ unterdrückt nur die Ausgabe dieser Zeile:

hello:
    @echo "Hallo!"

install:
    @echo "Installiere Abhängigkeiten …"
    npm ci

Mit just --quiet RECIPE kann die Befehlsanzeige für einen gesamten Aufruf abgeschaltet werden. Das unterdrückt nicht die eigentliche Programmausgabe.

Parameter verwenden

Pflichtparameter

# Eine Migration mit frei gewähltem Namen anlegen
migration name:
    npx prisma migrate dev --name {{ name }}
just migration add-users

Standardwerte

# Anwendung in einer Umgebung starten
run environment="development":
    npm run start -- --env {{ environment }}
just run
just run production

Mehrere Werte entgegennehmen

# Ausgewählte Tests starten; mindestens ein Wert ist erforderlich
test-files +files:
    pytest {{ files }}

# Git-Commit mit optionalen zusätzlichen Dateien
commit message *files:
    git add {{ files }}
    git commit -m "{{ message }}"

+parameter erwartet mindestens einen Wert, *parameter erlaubt auch null Werte. Werte mit Leerzeichen müssen beim Aufruf von der verwendeten Shell passend zitiert werden.

Variablen und Ausdrücke

Variablen werden einmal definiert und mit {{...}} in Recipe-Zeilen eingesetzt:

app := "meine-anwendung"
port := "8080"
image := "ghcr.io/beispiel/" + app

serve:
    python -m http.server {{ port }}

container:
    docker build -t {{ image }} .

Backticks führen bereits beim Auswerten des justfile einen Befehl aus und speichern dessen Standardausgabe:

revision := `git rev-parse --short HEAD`

info:
    @echo "Revision: {{ revision }}"

Das ist nützlich für Metadaten, sollte aber nicht für Befehle mit Seiteneffekten verwendet werden.

Umgebungsvariablen und .env

Mit env_var() werden vorhandene Umgebungsvariablen gelesen. env_var_or_default() liefert einen Ersatzwert, falls die Variable fehlt:

api_url := env_var_or_default("API_URL", "http://localhost:3000")

show-config:
    @echo "API: {{ api_url }}"

Um eine lokale .env-Datei automatisch einzulesen, wird die dotenv-Funktion aktiviert:

set dotenv-load

export DATABASE_URL := env_var("DATABASE_URL")

database-check:
    @echo "Prüfe Datenbankverbindung …"
    npx prisma db pull

export gibt eine just-Variable zusätzlich als Umgebungsvariable an die gestarteten Prozesse weiter. Geheimnisse gehören nicht direkt in das versionierte justfile; verwende dafür lokale .env-Dateien oder den Secret Store der CI-Plattform.

Abhängigkeiten zwischen Recipes

Abhängigkeiten hinter dem Doppelpunkt werden vor der eigentlichen Recipe ausgeführt:

lint:
    npm run lint

test:
    npm test

build:
    npm run build

# Führt zuerst lint, test und build aus
check: lint test build
    @echo "Alle Prüfungen erfolgreich."

# Deployment erst nach erfolgreichem Check
deploy: check
    ./scripts/deploy.sh

Mehrere Recipes können auch direkt nacheinander angefordert werden:

just lint test build

Ausführungsverzeichnis

just sucht vom aktuellen Ordner aus nach oben nach einem justfile und führt Recipes standardmäßig in dessen Verzeichnis aus. Dadurch funktioniert just test auch aus einem Unterordner des Projekts.

# Recipe in einem bestimmten Unterordner ausführen
[working-directory: "frontend"]
frontend:
    npm run dev

[working-directory: "backend"]
backend-test:
    pytest

Für ältere just-Versionen oder einfache Fälle ist ein explizites cd ebenfalls möglich. Jede normale Befehlszeile startet jedoch in einer eigenen Shell. Deshalb müssen zusammengehörige Befehle in derselben Zeile stehen:

# Falsch: Das zweite Kommando startet wieder im Projektverzeichnis
broken:
    cd frontend
    npm test

# Richtig
frontend-test:
    cd frontend && npm test

Windows und PowerShell

Unter Unix verwendet just standardmäßig sh. Unter Windows kann dieses beispielsweise durch Git for Windows bereitgestellt werden. Wer native PowerShell-Befehle verwenden möchte, setzt die Shell ausdrücklich im justfile:

# Windows PowerShell 5.1
set shell := ["powershell.exe", "-NoLogo", "-Command"]

files:
    Get-ChildItem -File

clean:
    Remove-Item -Recurse -Force -ErrorAction SilentlyContinue dist

Für PowerShell 7 wird stattdessen pwsh verwendet:

set shell := ["pwsh", "-NoLogo", "-NoProfile", "-Command"]

Soll ein justfile auf mehreren Betriebssystemen laufen, können plattformspezifische Shell-Einstellungen definiert werden:

set windows-shell := ["pwsh", "-NoLogo", "-NoProfile", "-Command"]
set shell := ["bash", "-cu"]

platform:
    @echo "Aktuelles Betriebssystem: {{ os() }}"

windows-shell gilt auf Windows, shell auf den übrigen Plattformen. Alle im Team benötigten Shells müssen installiert und über PATH erreichbar sein.

Mehrzeilige Scripts und andere Sprachen

Mit einer Shebang-Recipe wird der gesamte eingerückte Block als ein Script ausgeführt. Variablen und Kontrollstrukturen bleiben dadurch über alle Zeilen erhalten:

report:
    #!/usr/bin/env python3
    from pathlib import Path

    files = list(Path("src").rglob("*.py"))
    print(f"Python-Dateien: {len(files)}")

Eine portable Alternative ohne Unix-Shebang ist das [script]-Attribut:

[script("python3")]
report-json:
    import json
    print(json.dumps({"status": "ok"}))

Auch Parameter können an Script-Recipes übergeben werden:

[script("python3")]
greet name:
    import sys
    print(f"Hallo, {sys.argv[1]}!")

Private Recipes, Aliase und Bestätigung

Hilfs-Recipes ausblenden

_prepare:
    npm ci

dev: _prepare
    npm run dev

Namen mit führendem Unterstrich erscheinen nicht in der normalen Aufgabenliste, können aber weiterhin direkt oder als Abhängigkeit ausgeführt werden.

Kurze Aliase

alias t := test
alias b := build

test:
    npm test

build:
    npm run build

Gefährliche Aktionen bestätigen lassen

[confirm("Produktivsystem wirklich deployen?")]
deploy-production:
    ./scripts/deploy-production.sh

In nicht interaktiven CI-Läufen können bestätigungspflichtige Recipes mit --yes ausgeführt werden. Nutze dies nur in kontrollierten Pipelines.

Module und größere justfiles

Wächst die Datei stark an, lassen sich Recipes in Module auslagern. In justfile:

mod database
mod docker

Die Dateien database.just und docker.just enthalten die jeweiligen Recipes:

# database.just
migrate:
    npx prisma migrate deploy

seed:
    npx prisma db seed

Modul-Recipes werden mit dem Modulnamen aufgerufen:

just database migrate
just database seed

Mit import 'shared.just' werden Definitionen dagegen direkt in den aktuellen Namensraum importiert. Module sind bei größeren Projekten meist übersichtlicher, weil sie Namenskollisionen verhindern.

Praktisches vollständiges Beispiel

set dotenv-load

app := "shop-api"
port := env_var_or_default("PORT", "3000")

alias t := test
alias c := check

# Verfügbare Aufgaben anzeigen
default:
    @just --list

# Abhängigkeiten reproduzierbar installieren
install:
    npm ci

# Entwicklungsserver starten
dev:
    npm run dev -- --port {{ port }}

# Tests ausführen; optional kann ein Suchmuster angegeben werden
test pattern="":
    npm test -- {{ pattern }}

# Quellcode prüfen
lint:
    npm run lint

# Formatierung kontrollieren
format-check:
    npm run format:check

# Produktions-Build erzeugen
build:
    npm run build

# Gesamte lokale Qualitätsprüfung
check: lint format-check test build
    @echo "{{ app }} wurde erfolgreich geprüft."

# Container-Image mit Git-Revision bauen
image tag=`git rev-parse --short HEAD`: check
    docker build -t {{ app }}:{{ tag }} .

[confirm("Lokale Container und Volumes wirklich entfernen?")]
clean-docker:
    docker compose down --volumes --remove-orphans

Verwendung in GitHub Actions

In CI bleibt die eigentliche Befehlslogik im justfile. Die Pipeline muss dann nur just installieren und die passende Recipe starten:

name: CI

on:
  push:
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
      - uses: extractions/setup-just@v3
      - run: just install
      - run: just check

Damit lokale Entwicklung und CI wirklich dieselben Prüfungen ausführen, sollte die Pipeline bevorzugt vorhandene Recipes aufrufen statt deren Befehle erneut in YAML zu duplizieren.

Variablen beim Aufruf überschreiben

just --set port 9000 serve

Ausführung vorab kontrollieren

just --check
just --dry-run deploy

Shell-Autovervollständigung

PowerShell

Folgende Zeile kann in das PowerShell-Profil $PROFILE aufgenommen werden:

just --completions powershell | Out-String | Invoke-Expression

Bash

mkdir -p ~/.local/share/bash-completion/completions
just --completions bash > ~/.local/share/bash-completion/completions/just

Zsh

mkdir -p ~/.zsh/completions
just --completions zsh > ~/.zsh/completions/_just

Globale persönliche Recipes

Mit just -g wird ein globales justfile gesucht. Geeignete Speicherorte sind unter anderem ~/.config/just/justfile und ~/justfile:

# ~/.config/just/justfile

public-ip:
    curl https://ifconfig.me

git-clean-branches:
    git branch --merged | grep -vE "main|master"
just -g --list
just -g public-ip

Projektbezogene Recipes gehören weiterhin in das Repository. Das globale justfile eignet sich für persönliche, projektunabhängige Werkzeuge.

Fehlersuche

„No justfile found“

Im aktuellen Verzeichnis oder in dessen Elternordnern wurde kein justfile gefunden. In das Projekt wechseln oder die Datei ausdrücklich angeben:

just --justfile path/to/justfile --list

„Unknown recipe“

Recipe-Namen und verfügbare Module prüfen:

just --list
just --summary

Ein Befehl aus der vorherigen Zeile wirkt nicht weiter

Normale Recipe-Zeilen laufen in separaten Shell-Prozessen. Befehle mit && verbinden, [working-directory] verwenden oder eine Script-Recipe anlegen.

Ein Unix-Befehl funktioniert unter Windows nicht

Prüfen, welche Shell verwendet wird. Entweder Git Bash bereitstellen oder windows-shell beziehungsweise shell auf PowerShell setzen und die Befehle entsprechend schreiben.

Variablen werden falsch zerlegt

Interpolierte Werte mit Leerzeichen innerhalb der Recipe passend zitieren:

greet name:
    echo "Hallo {{ name }}"

Syntax prüfen

just --check
just --fmt --check

Empfohlene Konventionen

  • Eine hilfreiche default-Recipe an die erste Stelle setzen.
  • Öffentliche Recipes mit kurzen Kommentaren dokumentieren.
  • Namen wie dev, test, lint, build und check konsistent verwenden.
  • Destruktive Recipes durch klare Namen und [confirm] absichern.
  • Secrets über Umgebung oder Secret Stores statt über das justfile verteilen.
  • Plattformabhängige Shells ausdrücklich konfigurieren.
  • Mit just --check und just --fmt --check auch das justfile in CI prüfen.
  • Komplexe Logik in echte Scripts auslagern; Recipes bleiben der übersichtliche Einstiegspunkt.
