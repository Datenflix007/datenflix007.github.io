Aus Code eine Programm.exe erstellen

Ein praktischer Guide, wie du aus eigenen Programmen eine startbare Windows-EXE baust: mit Icon, Ressourcen, GUI-Modus, Konsolenmodus und sauberem Dist-Ordner.

Dieser Guide erklärt nicht zuerst, wie man einen Installer baut, sondern wie man überhaupt eine echte startbare Programm.exe erzeugt. Das ist der Schritt davor: Aus deinem Python-, Java-, C#-, Node.js-, Go- oder Rust-Projekt wird eine Datei oder ein App-Ordner, der auf einem Windows-PC ohne Entwicklungsumgebung gestartet werden kann.

Danach kannst du diese EXE optional mit dem Installer Guide weiterverpacken, damit sie im Startmenü auftaucht, eine Desktop-Verknüpfung bekommt und sauber deinstalliert werden kann.

EXE, App-Ordner und Installer

Diese drei Begriffe werden oft vermischt. Für einen sauberen Workflow solltest du sie trennen:

BegriffBedeutungBeispiel
Programm.exeStartbare Windows-Dateidist/MeineApp.exe
App-OrdnerEXE plus DLLs, Bilder, Daten und Runtime-Dateiendist/MeineApp/MeineApp.exe
InstallerSetup-Datei, die die App auf dem PC installiertMeineApp-Setup.exe

Eine einzelne EXE ist bequem zum Weitergeben. Ein App-Ordner ist oft stabiler, weil große Abhängigkeiten nicht in eine einzelne Datei gepackt werden müssen. Ein Installer ist dann sinnvoll, wenn normale Nutzerinnen und Nutzer die App dauerhaft installieren sollen.

Empfohlener Ablauf

  1. Projekt lokal normal starten und Fehler beheben.
  2. Alle benötigten Dateien sammeln: Icons, Bilder, JSON, Datenbanken, Modelle.
  3. Entscheiden, ob es eine CLI-App oder GUI-App ist.
  4. Mit dem passenden Tool eine EXE oder einen App-Ordner bauen.
  5. Die erzeugte App auf einem möglichst sauberen Windows-PC testen.
  6. Erst danach optional einen Installer bauen.

Der wichtigste Test lautet: Funktioniert die App auch, wenn du sie aus dem dist/-Ordner startest und nicht aus deiner IDE?

Python: EXE mit PyInstaller

Für Python-Projekte ist PyInstaller der einfachste Einstieg. Es nimmt dein Startskript, analysiert Imports, sammelt Abhängigkeiten ein und erzeugt daraus eine EXE.

CLI-Programm

python -m venv .venv
.venv\Scripts\activate
pip install pyinstaller

pyinstaller --onefile --name MeineApp main.py

Das Ergebnis liegt danach normalerweise unter:

dist\MeineApp.exe

Tkinter- oder GUI-Programm

Bei Tkinter, PySide, PyQt oder CustomTkinter willst du meistens kein schwarzes Konsolenfenster. Dafür nutzt du --windowed:

pyinstaller --onefile --windowed ^
  --name MeineTkApp ^
  --icon assets\app.ico ^
  main.py

Das Icon muss unter Windows am besten eine echte .ico-Datei sein. PNG-Dateien funktionieren hier nicht immer direkt.

Ressourcen mitgeben

Wenn deine App Dateien braucht, musst du sie explizit mitgeben. Typische Beispiele sind Bilder, Templates, JSON-Dateien, SQLite-Datenbanken oder ML-Modelle.

pyinstaller --onefile --windowed ^
  --name Stundenplan ^
  --icon assets\stundenplan.ico ^
  --add-data "assets;assets" ^
  --add-data "data;data" ^
  main.py

Im Code solltest du Ressourcen nicht nur relativ zu deinem Projektordner suchen, sondern auch im entpackten PyInstaller-Ordner. Eine robuste Hilfsfunktion sieht so aus:

from pathlib import Path
import sys

def resource_path(relative):
    if hasattr(sys, "_MEIPASS"):
        return Path(sys._MEIPASS) / relative
    return Path(__file__).parent / relative

logo = resource_path("assets/logo.png")

Ordner statt Onefile

--onefile ist bequem, startet aber manchmal langsamer, weil die App sich erst temporär entpackt. Für größere Programme ist ein App-Ordner oft besser:

pyinstaller --windowed --name MeineApp --icon assets\app.ico main.py

Dann liegt die App unter dist\MeineApp\MeineApp.exe zusammen mit allen benötigten Dateien.

Python: Alternative mit Nuitka

Nuitka kompiliert Python-Code stärker in Richtung C/C++-Build und kann für manche Projekte schneller oder robuster sein. Dafür ist die Einrichtung etwas anspruchsvoller.

pip install nuitka

python -m nuitka ^
  --standalone ^
  --windows-console-mode=disable ^
  --windows-icon-from-ico=assets\app.ico ^
  --output-filename=MeineApp.exe ^
  main.py

Nutze Nuitka, wenn PyInstaller bei deinem Projekt Probleme macht oder wenn du bewusst einen kompilierten Build testen willst.

Java: EXE mit jpackage

Für moderne Java-Projekte ist jpackage der saubere Weg. Es kommt mit dem JDK und kann aus deiner Java-App einen nativen Windows-App-Ordner oder direkt ein Installer-Paket bauen. Hier geht es zuerst um die App bzw. EXE.

Jar vorbereiten

mvn clean package

# oder bei Gradle:
gradlew build

Wichtig ist, dass deine JAR-Datei eine Main-Class hat und lokal startet:

java -jar target\meineapp.jar

App-Image mit EXE bauen

jpackage ^
  --type app-image ^
  --name MeineJavaApp ^
  --input target ^
  --main-jar meineapp.jar ^
  --main-class de.beispiel.Main ^
  --icon assets\app.ico ^
  --dest dist

Danach entsteht ein Ordner wie dist\MeineJavaApp\. Darin liegt eine MeineJavaApp.exe und meist eine passende Java Runtime. Nutzer müssen dann nicht separat Java installieren.

Wenn du nur eine JAR hast

Eine .jar ist nicht dasselbe wie eine .exe. Wenn du nur eine JAR verteilst, braucht der PC Java. Wenn du eine richtige Windows-App willst, nutze jpackage oder einen Launcher wie Launch4j.

C#: EXE mit dotnet publish

Bei C# und .NET ist der Standardweg dotnet publish. Damit kannst du eine App mit oder ohne installierte .NET Runtime bauen.

Normale EXE bauen

dotnet publish -c Release -r win-x64 --self-contained false

Das ist klein, setzt aber voraus, dass die passende .NET Runtime installiert ist.

Self-contained Single-File

dotnet publish -c Release -r win-x64 ^
  --self-contained true ^
  -p:PublishSingleFile=true ^
  -p:IncludeNativeLibrariesForSelfExtract=true

Das Ergebnis ist größer, kann aber auf mehr Windows-Systemen direkt laufen.

Node.js: CLI als EXE bauen

Wenn dein Projekt ein Node.js-CLI-Tool ist, kannst du es mit Tools wie pkg oder nexe als EXE verpacken. Für grafische Desktop-Apps ist Electron meist passender.

npm install
npm install -D pkg

npx pkg . --targets node18-win-x64 --output dist/meine-cli.exe

Für Electron-Apps erzeugst du normalerweise keinen einfachen Ein-Datei-Build, sondern einen App-Ordner:

npm install -D electron-builder
npx electron-builder --win portable

Go und Rust: native EXE direkt bauen

Go und Rust erzeugen von Haus aus native Programme. Hier brauchst du oft kein extra Verpackungstool.

Go

go build -o dist\MeineGoApp.exe .

Rust

cargo build --release

Die Rust-EXE liegt danach zum Beispiel unter target\release\meine_app.exe.

Icon, Version und Dateiinformationen

Eine EXE wirkt deutlich fertiger, wenn sie ein Icon und sinnvolle Metadaten hat. Das betrifft vor allem Windows Explorer, Taskleiste und SmartScreen-Wahrnehmung.

  • Icon: Unter Windows am besten .ico mit mehreren Größen, zum Beispiel 16, 32, 48 und 256 Pixel.
  • Name: Der Dateiname sollte eindeutig sein, etwa Stundenplan.exe statt main.exe.
  • Version: Nutze Versionen wie 1.0.0, damit Downloads nachvollziehbar bleiben.
  • Signatur: Für öffentliche Windows-Downloads ist Code Signing langfristig sinnvoll, aber für kleine Lernprojekte nicht zwingend.

Interaktives Windows-Buildskript

Dieses einfache .bat-Skript fragt die wichtigsten Werte ab und baut je nach Projekttyp eine EXE. Es ist bewusst als Startpunkt gedacht.

@echo off
setlocal

echo === Programm.exe Build-Assistent ===
set /p APP_NAME=App-Name:
set /p STACK=Typ python-cli/python-gui/java/csharp/go/rust:
set /p MAIN_FILE=Startdatei oder Projektpfad:
set /p ICON=Iconpfad optional:

if "%STACK%"=="python-cli" (
  pyinstaller --onefile --name "%APP_NAME%" "%MAIN_FILE%"
)

if "%STACK%"=="python-gui" (
  if "%ICON%"=="" (
    pyinstaller --onefile --windowed --name "%APP_NAME%" "%MAIN_FILE%"
  ) else (
    pyinstaller --onefile --windowed --name "%APP_NAME%" --icon "%ICON%" "%MAIN_FILE%"
  )
)

if "%STACK%"=="java" (
  set /p JAR=Pfad zur JAR:
  set /p MAIN_CLASS=Main-Class:
  jpackage --type app-image --name "%APP_NAME%" --input "%MAIN_FILE%" --main-jar "%JAR%" --main-class "%MAIN_CLASS%" --dest dist
)

if "%STACK%"=="csharp" (
  dotnet publish "%MAIN_FILE%" -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true
)

if "%STACK%"=="go" (
  go build -o "dist\%APP_NAME%.exe" "%MAIN_FILE%"
)

if "%STACK%"=="rust" (
  cargo build --release
)

echo.
echo Fertig. Schau in dist\, target\release\ oder den Publish-Ordner.
pause

Linux-Hinweis

Eine Windows-EXE ist für Windows gedacht. Unter Linux baust du normalerweise keine .exe, sondern ein ausführbares Binary, ein AppImage, ein .deb-Paket oder eine .desktop-Integration. Für reine CLI-Tools reicht oft:

# Go
go build -o dist/meine-app .

# Rust
cargo build --release

# Python unter Linux mit PyInstaller
pyinstaller --onefile --name meine-app main.py

Wenn du unter Linux eine Windows-EXE bauen willst, brauchst du Cross-Compilation oder eine Windows-Buildumgebung. Für Python ist das besonders unpraktisch: PyInstaller sollte am besten auf dem Zielsystem gebaut werden, also Windows-Build auf Windows, Linux-Build auf Linux.

Prompt für einen Coding Agent

Wenn du ein bestehendes Projekt automatisch in einen baubaren EXE-Workflow umbauen lassen willst, kannst du diesen Prompt verwenden:

Analysiere dieses Projekt und richte einen reproduzierbaren Windows-EXE-Build ein.

Ziele:
- Erkenne Sprache, Framework, Einstiegspunkt und benötigte Ressourcen.
- Erzeuge oder aktualisiere ein Buildskript für Windows.
- Die fertige App soll unter dist/ liegen.
- Falls es eine GUI-App ist, soll kein Konsolenfenster erscheinen.
- Falls Icons oder Assets vorhanden sind, sollen sie eingebunden werden.
- Dokumentiere im README kurz, wie man die EXE baut und testet.
- Verändere die Programmlogik nur, wenn es für den Build zwingend nötig ist.

Bitte erkläre danach:
1. Welches Packaging-Tool du gewählt hast.
2. Welche Dateien neu sind.
3. Welcher Befehl die EXE baut.
4. Wo die fertige EXE liegt.
5. Welche Einschränkungen noch gelten.

Wann danach ein Installer kommt

Eine Programm.exe reicht für kleine Tools, Tests, Schulprojekte oder interne Skripte oft aus. Sobald die App aber dauerhaft auf fremden PCs genutzt werden soll, brauchst du meistens den nächsten Schritt:

  • Startmenü-Eintrag
  • Desktop-Verknüpfung
  • Installationspfad
  • Uninstaller
  • Update- oder Versionskonzept
